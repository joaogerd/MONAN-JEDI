# MONAN-JEDI YAML configuration reference

This document explains the meaning of each entry in `config/jaci.yaml`, how to modify it, and what behavior to expect after each change.

The YAML file is the normal user-editable interface for the MONAN-JEDI workflow. The shell scripts should not need to be edited for routine use.

## 1. General model

The workflow separates three concepts:

1. `MONAN-JEDI` repository root: source tree and bundle definition. The top-level `CMakeLists.txt` is the project-controlled bundle definition.
2. `project.root`: user-owned workspace where build trees and logs are created.
3. `stack.*`: existing spack-stack installation. This may be owned by another user and may be read-only for normal users.

The workflow no longer clones `JCSDA/jedi-bundle` and no longer applies a CMakeLists template during the build.

## 2. `site`

```yaml
site: jaci
```

Defines the target site name. The current expected value is `jaci`.

Changing this value does not automatically make the workflow portable to another machine. Site-specific module loading is still implemented for JACI in `scripts/lib/stack.sh` and through `stack.site_setup`.

## 3. `project.root`

```yaml
project:
  root: /p/projetos/monan_das/${USER}
```

Defines the user-owned MONAN-JEDI workspace root.

Derived directories:

```text
${project.root}/work/${build.id}
${project.root}/logs/${build.id}
```

Mapped internal variable:

```text
PROJECT_ROOT
```

Use a writable shared filesystem path on JACI, preferably under `/p` or `/lustre`, because PBS compute nodes need to access the same files.

## 4. `stack` block

The `stack` block describes the existing spack-stack installation consumed by MONAN-JEDI.

### `stack.owner`

```yaml
stack:
  owner: joao.gerd
```

Unix account that owns the spack-stack installation.

Mapped internal variable:

```text
STACK_OWNER
```

Use your own username if you built your own stack. Use another account if consuming a shared stack maintained by that account.

### `stack.instance`

```yaml
stack:
  instance: spack-stack-inpe-overlay-20260515T181917Z
```

Identifier of the spack-stack instance generated by `spack-stack-inpe`.

Mapped internal variable:

```text
STACK_INSTANCE
```

Used to derive:

```text
/p/projetos/monan_das/${stack.owner}/work/${stack.instance}
```

### `stack.work_root`

```yaml
stack:
  work_root:
```

Optional explicit path to the stack work root.

Mapped internal variable:

```text
STACK_WORK_ROOT
```

If empty, the scripts derive:

```text
/p/projetos/monan_das/${stack.owner}/work/${stack.instance}
```

### `stack.root`

```yaml
stack:
  root:
```

Optional explicit path to the root of the spack-stack checkout or installation.

Mapped internal variable:

```text
STACK_ROOT
```

If empty, the scripts derive:

```text
${stack.work_root}/spack-stack
```

### `stack.env_name`

```yaml
stack:
  env_name: jaci-mpas-jedi-gcc12-craympich
```

Name of the generated spack-stack environment.

Mapped internal variable:

```text
STACK_ENV_NAME
```

Used to derive:

```text
${stack.root}/envs/${stack.env_name}/modules
```

### `stack.module_root`

```yaml
stack:
  module_root:
```

Optional explicit path to generated module files.

Mapped internal variable:

```text
STACK_MODULE_ROOT
```

If empty, the scripts derive:

```text
${stack.root}/envs/${stack.env_name}/modules
```

### `stack.site_setup`

```yaml
stack:
  site_setup: configs/sites/tier2/jaci/setup.sh
```

Site setup script inside the spack-stack tree.

Mapped internal variable:

```text
STACK_SITE_SETUP
```

On JACI this should point to the site setup inside the spack-stack tree, not the root `spack-stack/setup.sh`.

### `stack.env_module`

```yaml
stack:
  env_module: cray-mpich/8.1.31/none/none/jedi-mpas-env/1.0.0
```

Generated spack-stack environment module loaded after `module use ${stack.module_root}`.

Mapped internal variable:

```text
STACK_ENV_MODULE
```

This controls the dependency set visible to CMake, ecbuild, make and ctest.

## 5. `build` block

### `build.id`

```yaml
build:
  id: monan-jedi-mpas-only
```

Identifier of one MONAN-JEDI build/test attempt.

Mapped internal variable:

```text
MONAN_JEDI_RUN_ID
```

Used to derive:

```text
${project.root}/work/${build.id}
${project.root}/logs/${build.id}
```

The configured build directory is derived as:

```text
${project.root}/work/${build.id}/build
```

### `build.jobs`

```yaml
build:
  jobs: 8
```

Number of parallel jobs passed to `make -j`.

Mapped internal variable:

```text
MONAN_JEDI_BUILD_JOBS
```

Increase only if the node and filesystem can handle it. Decrease if builds fail due to memory pressure or filesystem stress.

## 6. `model` block

The `model` block controls MPAS-specific CMake configuration options used during `ecbuild`.

### `model.double_precision`

```yaml
model:
  double_precision: ON
```

Controls the CMake option passed to `mpas-bundle`:

```text
MPAS_DOUBLE_PRECISION
```

Mapped internal variable:

```text
MONAN_JEDI_MODEL_DOUBLE_PRECISION
```

Accepted values:

```text
ON
OFF
```

Use `ON` when validating the build with the `mpas-jedi` CTest suite. This is the upstream default and matches the precision used to produce the upstream CTest reference files.

Use `OFF` only when the target workflow or tutorial requires a single-precision MPAS build, for example MPAS-Workflow calculations using the `mpas-bundle` build.

This distinction is important because the MPAS-JEDI tutorial documents that CTest reference files are produced with a double-precision build. Therefore, some CTest cases may fail when the bundle is compiled in single precision because the numerical difference can be larger than the test tolerance.

Recommended validation setting:

```yaml
model:
  double_precision: ON
```

Single-precision workflow setting:

```yaml
model:
  double_precision: OFF
```

Do not interpret CTest failures from a single-precision build as MPI, compiler, filesystem, or platform failures without first checking the detailed CTest logs.

## 7. `compilers` block

```yaml
compilers:
  cc: cc
  cxx: CC
  fc: ftn
  f77: ftn
  f90: ftn
```

Compiler commands or absolute compiler wrapper paths used by CMake/ecbuild.

Mapped internal variables:

```text
MONAN_JEDI_CC
MONAN_JEDI_CXX
MONAN_JEDI_FC
MONAN_JEDI_F77
MONAN_JEDI_F90
```

Resolved runtime variables:

```text
CC
CXX
FC
F77
F90
```

Recommended JACI values are `cc`, `CC` and `ftn`. Avoid raw `gcc`, `g++` or `gfortran` on JACI unless you are intentionally bypassing CrayPE.

## 8. `mpi` block

```yaml
mpi:
  mpicc: cc
  mpicxx: CC
  mpifc: ftn
  mpif77: ftn
  mpif90: ftn
```

MPI compiler wrapper commands or absolute paths.

Mapped internal variables:

```text
MONAN_JEDI_MPICC
MONAN_JEDI_MPICXX
MONAN_JEDI_MPIFC
MONAN_JEDI_MPIF77
MONAN_JEDI_MPIF90
```

Resolved runtime variables:

```text
MPICC
MPICXX
MPIFC
MPIF77
MPIF90
```

On CrayPE, `cc`, `CC` and `ftn` act as compiler and MPI-aware wrappers.

## 9. `ctest` block

### `ctest.login_regex`

```yaml
ctest:
  login_regex: "^mpasjedi_coding_norms$"
```

CTest regex used by the login-node-safe test command.

Mapped internal variable:

```text
MONAN_JEDI_CTEST_REGEX
```

Use a narrow regex for tests known to be safe on login nodes.

### `ctest.pbs_regex`

```yaml
ctest:
  pbs_regex: ""
```

Optional CTest regex for PBS tests.

Mapped internal variable:

```text
MONAN_JEDI_CTEST_PBS_REGEX
```

Current full-suite policy: keep this empty to avoid restricting tests with `ctest -R`.

### `ctest.exclude_regex`

```yaml
ctest:
  exclude_regex: "^(ioda_bufr_python_encoder|ioda_bufr_python_parallel|mpasjedi_lgetkf_height_vloc)$"
```

CTest exclusion regex used by PBS full-suite testing.

Mapped internal variable:

```text
MONAN_JEDI_CTEST_EXCLUDE_REGEX
```

Add known failing tests only after confirming they are unrelated to the current build or environment change.

### `ctest.jobs`

```yaml
ctest:
  jobs: 1
```

Parallelism passed to `ctest -j`.

Mapped internal variable:

```text
MONAN_JEDI_CTEST_JOBS
```

Start with `1` for first validation on JACI.

### `ctest.allow_login_node_mpi_tests`

```yaml
ctest:
  allow_login_node_mpi_tests: false
```

Safety flag documenting whether MPI tests may be run on login nodes.

Mapped internal variable:

```text
ALLOW_LOGIN_NODE_MPI_TESTS
```

Keep `false` on JACI for normal use.

## 10. `pbs` block

### `pbs.queue`

```yaml
pbs:
  queue: pesqmini
```

PBS queue used for compute-node CTest jobs.

Mapped internal variable:

```text
MONAN_JEDI_PBS_QUEUE
```

### `pbs.ncpus`

```yaml
pbs:
  ncpus: 64
```

Number of CPUs requested for the PBS job.

Mapped internal variable:

```text
MONAN_JEDI_PBS_NCPUS
```

### `pbs.walltime`

```yaml
pbs:
  walltime: "00:30:00"
```

Maximum wall-clock time requested for the PBS job.

Mapped internal variable:

```text
MONAN_JEDI_PBS_WALLTIME
```

Increase for full validations when necessary.

### `pbs.submit_job`

```yaml
pbs:
  submit_job: true
```

Controls whether the PBS script is submitted automatically after generation.

Mapped internal variable:

```text
MONAN_JEDI_SUBMIT_JOB
```

Use `false` when reviewing the generated PBS script before submission.

## 11. Recommended current JACI testing configuration

For a full PBS validation excluding the three known failing tests:

```yaml
ctest:
  login_regex: "^mpasjedi_coding_norms$"
  pbs_regex: ""
  exclude_regex: "^(ioda_bufr_python_encoder|ioda_bufr_python_parallel|mpasjedi_lgetkf_height_vloc)$"
  jobs: 1
  allow_login_node_mpi_tests: false

pbs:
  queue: pesqmini
  ncpus: 64
  walltime: "02:00:00"
  submit_job: true
```

## 12. Environment-variable override behavior

The YAML values are read by `scripts/lib/read_config.py`. If an environment variable is already exported before running the workflow, that value takes precedence over the YAML-derived value.

Example:

```bash
export MONAN_JEDI_CTEST_JOBS=2
bash scripts/monan-jedi.sh test-pbs --config config/jaci.yaml
```

Use this only for temporary tests. For reproducible runs, prefer editing the YAML and committing the change.
