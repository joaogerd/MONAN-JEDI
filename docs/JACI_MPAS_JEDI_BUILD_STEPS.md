# JACI MPAS-JEDI build steps

## Purpose

This document describes the initial MONAN-JEDI workflow for building a reduced MPAS-JEDI-only JEDI bundle on JACI.

The workflow depends on a previously built and validated `spack-stack-inpe` environment.

## Confirmed prerequisite

Before using this repository, the JACI stack must have completed the stack creation, package installation, module generation and validation steps from `spack-stack-inpe`.

The MONAN-JEDI scripts do not create or maintain the stack. They consume the generated modules from an existing stack.

## Main variables

```bash
export STACK_OWNER="joao.gerd"
export STACK_TEST_ID="spack-stack-inpe-overlay-20260515T181917Z"
export MONAN_JEDI_TEST_ID="monan-jedi-mpas-only-$(date -u +%Y%m%dT%H%M%SZ)"
```

`STACK_OWNER` identifies the Unix account that owns the validated stack. If the user owns the stack, this can be omitted because the default is `${USER}`.

`STACK_TEST_ID` identifies the validated stack built by `spack-stack-inpe`.

`MONAN_JEDI_TEST_ID` identifies this MPAS-JEDI build attempt.

Optional user workspace override:

```bash
export PROJECT_ROOT="/p/projetos/monan_das/${USER}/SPACK/jgerd"
```

`PROJECT_ROOT` controls where MONAN-JEDI writes work trees and logs. It does not control where the shared stack is located unless `STACK_OWNER`, `STACK_WORK_ROOT` or `STACK_ROOT` are left to defaults.

## User-mode stack loading

The user-mode interface to a shared stack is based on generated modules:

```bash
source "${STACK_ROOT}/configs/sites/tier2/jaci/setup.sh"
module use "${STACK_MODULE_ROOT}"
module load "${STACK_ENV_MODULE}"
```

The MONAN-JEDI workflow must not source:

```bash
source "${STACK_ROOT}/setup.sh"
```

The root `setup.sh` belongs to the spack-stack administration workflow. It can set `SPACK_STACK_DIR`, enable spack-stack shell completions, source Spack itself and change the Spack bootstrap path. In a shared tree owned by another user, that can trigger writes under `${STACK_ROOT}/spack` and can also expose Git `dubious ownership` checks.

## Workflow

```bash
cd /p/projetos/monan_das/${USER}/projects/MONAN-JEDI

git pull

export STACK_OWNER="joao.gerd"
export STACK_TEST_ID="spack-stack-inpe-overlay-20260515T181917Z"
export MONAN_JEDI_TEST_ID="monan-jedi-mpas-only-$(date -u +%Y%m%dT%H%M%SZ)"

bash scripts/01_load_stack_env.sh
bash scripts/02_prepare_jedi_bundle.sh
bash scripts/03_create_mpas_only_bundle.sh
bash scripts/04_configure_mpas_jedi.sh
bash scripts/05_build_mpas_jedi.sh
bash scripts/06_test_mpas_jedi.sh
bash scripts/07_collect_logs.sh
```

If the MONAN-JEDI checkout, work tree and logs should live under another directory:

```bash
export PROJECT_ROOT="/p/projetos/monan_das/${USER}/SPACK/jgerd"
```

## What each script does

### `01_load_stack_env.sh`

Loads the Tcl module generated by the validated `spack-stack-inpe` environment and validates Python, `mpi4py` and `netCDF4`.

This script loads the stack in user mode through the JACI site setup and generated modules. It must not source the root `setup.sh` from the shared `spack-stack` tree.

### `02_prepare_jedi_bundle.sh`

Clones or updates `JCSDA/jedi-bundle`, checks out the selected branch or tag and initializes submodules.

This script does not load the stack because it only needs Git and writes only to the user-owned source tree.

### `03_create_mpas_only_bundle.sh`

Replaces the upstream `jedi-bundle/CMakeLists.txt` with a reduced MPAS-JEDI-only bundle used for JACI compatibility validation.

This script does not load the stack because it only edits the user-owned source tree.

### `04_configure_mpas_jedi.sh`

Configures the reduced bundle with `ecbuild`, using CrayPE compiler drivers `cc`, `CC` and `ftn`.

This script loads the stack in user mode because `ecbuild`, CMake dependencies and JEDI libraries are provided by the generated environment module.

### `05_build_mpas_jedi.sh`

Builds the configured bundle using `make`.

### `06_test_mpas_jedi.sh`

Runs CTest. Tests may require compute-node execution depending on the test.

### `07_collect_logs.sh`

Collects and summarizes logs.

## Important technical decision

The MPAS-JEDI workflow does not create or modify the software stack. It only consumes the stack generated by `spack-stack-inpe`.

This separation keeps site configuration and JEDI/MONAN build logic independent. It also allows several users to consume the same stack without write permission to the stack tree.

## Quick validation

After `01_load_stack_env.sh`, the log should not contain these messages:

```text
Setting environment variable SPACK_STACK_DIR
Enabling shell completions for spack-stack extensions
Updating file: spack-completion.bash
Sourcing spack environment
Changing bootstrap path
fatal: detected dubious ownership
```

Those messages indicate that the root `spack-stack/setup.sh` was sourced, which is not expected in this user workflow.

The environment log should show the selected stack and module paths, for example:

```text
STACK_ROOT=/p/projetos/monan_das/joao.gerd/work/spack-stack-inpe-overlay-20260515T181917Z/spack-stack
STACK_MODULE_ROOT=.../envs/jaci-mpas-jedi-gcc12-craympich/modules
STACK_ENV_MODULE=cray-mpich/8.1.31/none/none/jedi-mpas-env/1.0.0
```

## Logs

By default logs are written under:

```text
/p/projetos/monan_das/${USER}/logs/${MONAN_JEDI_TEST_ID}
```

Work trees are written under:

```text
/p/projetos/monan_das/${USER}/work/${MONAN_JEDI_TEST_ID}
```

If `PROJECT_ROOT` is set, both paths are placed under that directory:

```text
${PROJECT_ROOT}/logs/${MONAN_JEDI_TEST_ID}
${PROJECT_ROOT}/work/${MONAN_JEDI_TEST_ID}
```
